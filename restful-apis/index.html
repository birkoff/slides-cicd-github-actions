RESTful API Design Guidelines

Principles for Scalable & Evolvable Systems

Based on the insights of Hector Reyes-Aleman

The Core Goal

  • Predictability & Intuition
  • Stability over "Cleanliness"
  • Client-Safe Evolvability
Focus on long-term maintenance rather than just initial setup.

1. Resources Over Actions

Use Nouns, not Verbs in your endpoints.

❌ Bad (RPC)

POST /processInvoice
POST /startBatch

✅ Good (REST)

POST /invoices
GET /batches

2. Hierarchical Relationships

The path should describe the domain structure.


                    GET /api/projects/{id}/documents

This clearly communicates: "Documents belonging to a specific Project."

3. Identification vs. Filtering

Type Use Case Example
Path Identity /users/123
Query Refinement ?sort=desc

4. Flatten Your API

Limit nesting to 1 or 2 levels deep.

❌ /projects/1/batches/2/docs/9/pages/3

✅ /documents/9/pages/3

Deep nesting makes URLs brittle and hard to change.

5. Model State Transitions

Instead of custom "action" endpoints, update the resource state.


                    // Update state, don't trigger "verbs"
                    PATCH /api/invoices/100
                    {
                      "status": "approved"
                    }

6. Idempotency

Retrying a request should be safe.

  • GET, PUT, DELETE: Inherently idempotent.
  • POST: Use Idempotency-Key headers to prevent duplicate records.

7. Semantic Status Codes

  • 201 Created - Success + New Resource
  • 400 Bad Request - Client error
  • 404 Not Found - Missing resource
  • 429 Too Many Requests - Rate limit

8. Modern Pagination

Prefer Cursor-based over Page-based.

GET /api/docs?cursor=Ym9va21hcms&limit=25

Naming Standards

  • Plural Nouns: /users
  • Kebab-case URLs: /user-profiles
  • No Abbreviations: /documents (not /docs)
  • Consistency: Choose snake_case or camelCase and stick to it.

Final Word

"An API is a public contract. Design for the developer who will use it three years from now."

3. Query Params vs. Paths

  • Path Segments: Used for identification.
  • Query Params: Used for filtering, sorting, and pagination.
/* Identification */
                GET /api/invoices/100

                /* Filtering & Sorting */
                GET /api/invoices?status=paid&sort=-created_at
