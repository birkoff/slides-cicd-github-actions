Context and Problem Statement

Currently, the team encounters two main sources of friction during the sprint lifecycle:

Premature Implementation

Development often begins before the full scope or edge cases of a ticket are understood, leading to mid-sprint context switching and scope creep.

Integration Delays

FE and BE developers often work in silos. When the two layers finally integrate, mismatches in data types, payload structures, or HTTP status codes cause "integration hell," resulting in rework and blocked tickets.

The Decision

We are adopting a strict "Analyze First, Contract First" policy. This consists of two mandatory agreements...

1. The "First 3 Days" Analysis Rule

Developers must read assigned tickets, validate understanding of requirements, Story points accuracy and identify ambiguity.

2. Mandatory Schema Contracts (FE/BE Handshake)

Before any code is written for a feature involving data exchange, a Contract Schema Agreement must be defined. This contract implies that the Interface is agreed upon before the Implementation details are worked out.

This agreement must be documented and must explicitly cover the following four quadrants:

Quadrant Description Technical Detail Required
API Request/Response Schemas Validation rules, Required vs. Optional fields, Data Types.
Data Data table Schemas Initial Layout.

Consequences

Positive Outcomes:

  • Reduced Rework: Clarifying requirements in the first 3 days prevents coding the "wrong thing."
  • Parallel Development: Once the Contract Schema is agreed upon, FE can mock the data exactly as agreed, and BE can build the implementation. Neither blocks the other.
  • Documentation: The schema agreement acts as live documentation for the feature.

Negative Outcomes / Risks:

  • Perceived Slowness: The velocity may appear slower in the first few days of the sprint because less code is being committed.
  • Discipline Required: It requires strict discipline to pause and document the contract before rushing to solve the problem in code.

Compliance

  • Pull Requests (PRs) involving client-server communication should link to the agreed Contract Schema.
  • During Stand-up on Day 4, the Scrum Master/Lead will verify that all active tickets have an established contract.
