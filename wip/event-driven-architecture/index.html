API Services & Event-Driven Communication Rules

Based on the article by Hector Reyes Aleman

1. Domain Services

Encapsulates pure domain logic that doesn't fit in a single entity.

Communication Rules

DO Call ✅

  • Entities / Value Objects
  • Other Domain Services
  • Repositories (via protocols)

DON'T Call ❌

  • Application Services
  • Infrastructure code
  • External Systems / SDKs

2. Application Services

Orchestrates workflows and connects domain logic with infrastructure.

Communication Rules

  • Allowed: Call Domain Services, Entities, and Infrastructure (via interfaces).
  • Forbidden: Direct infrastructure calls or reaching into another context's internal logic.

3. Communication Direction


                    Application Services -> Domain Services (OK)
                    Domain Services -> Domain Entities (OK)

                    Domain Services -> Application Services (WRONG)
                    Domain Logic -> Infrastructure Code (WRONG)

4. Bounded Contexts

Communication between contexts must happen via contracts, never via internal models.

Do NOT Share:

  • Domain Entities
  • Databases
  • Internal Domain Events

Use: APIs (REST/gRPC) or Published Events.

5. The Pub/Sub Pattern

Enables asynchronous, decoupled communication.

  • Publisher: Emits events; unaware of consumers.
  • Subscriber: Attaches rules to filter/consume events.

Example: Document Pipeline

Services involved: Process, Processed, Export, Notifications.

The Publisher Lambda


import boto3
client = boto3.client("events")

def handler(event, context):
    entry = {
        "Source": "document.process",
        "DetailType": "Document.Processed",
        "Detail": json.dumps({"documentId": "123", "status": "SUCCESS"})
    }
    client.put_events(Entries=[entry])

EventBridge Rule (YAML)


EventPattern:
  source:
    - document.process
  detail-type:
    - Document.Processed
State: ENABLED

Key Takeaways

  1. Keep Domain Services pure.
  2. Orchestrate in Application Services.
  3. Publisher has NO routing logic.
  4. Subscriber defines its own interest.
